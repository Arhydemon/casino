# Техническое задание

## Проект: кроссплатформенная игровая платформа на Python/Flet

## 1. Общая информация

**Название проекта:** клиент-серверная игровая платформа с набором карточных и казино-игр.

**Назначение проекта:** разработка учебного приложения для экзамена по ООП.

**Формат проекта:** desktop-приложение для Windows и Android APK с единым интерфейсом, общей системой аккаунтов, баланса, статистики и игровых комнат.

**Команда:** 2 разработчика.

**Срок разработки:** до 2 месяцев.

**Основная цель:** создать архитектурно грамотное приложение с использованием принципов ООП, модульной структуры, клиент-серверного взаимодействия и сохранения пользовательских данных.

---

## 2. Цели и задачи проекта

### 2.1. Цель

Разработать кроссплатформенное приложение, в котором пользователь может:

- зарегистрироваться и войти в аккаунт;
- играть в несколько игр;
- использовать локальный одиночный режим без сервера;
- использовать сетевой режим по локальной сети через временный сервер на ноутбуке;
- создавать и подключаться к игровым комнатам;
- сохранять баланс и статистику;
- использовать современный минималистичный интерфейс с анимациями и звуковыми эффектами.

### 2.2. Основные задачи

- реализовать общую архитектуру приложения;
- реализовать систему пользователей и авторизации;
- реализовать локальное хранение и серверное взаимодействие;
- реализовать игровые модули;
- реализовать систему комнат;
- реализовать сохранение баланса и статистики;
- реализовать UI на Flet;
- подготовить проект к запуску на Windows и сборке Android APK.

## 5. Функциональные требования

## 5.1. Система пользователей

Приложение должно поддерживать:

- регистрацию пользователя;
- вход по username и password;
- сохранение сессии пользователя;
- автоматический вход при повторном запуске приложения;
- безопасное хранение пароля в виде хеша.

### 5.1.1. Поля пользователя

- username
- password_hash
- дата регистрации

### 5.1.2. Требования к входу

- после успешной регистрации пользователь не должен вводить логин и пароль каждый раз при запуске приложения;
- должна быть локально сохранённая авторизационная сессия.

---

## 5.2. Профиль пользователя

Для каждого пользователя должны храниться:

- общий баланс фишек;
- количество сыгранных игр;
- количество побед;
- общее количество выигранных фишек.

### 5.2.1. Баланс

- баланс общий для всех игр;
- пользователь может уйти в ноль;
- при нулевом или недостаточном балансе должно быть доступно пополнение по кнопке.

### 5.2.2. Статистика

Система должна сохранять:

- общее количество игр;
- общее количество побед;
- сумму общего выигрыша.

История отдельных матчей в первой версии не требуется.

---

## 5.3. Игры

В состав платформы входят:

- Слоты
- Очко
- Рулетка
- Дурак
- Покер Texas Hold’em

### 5.3.1. Распределение по разработчикам

**Разработчик 1:** Дурак, Покер

**Разработчик 2:** Слоты, Очко, Рулетка

---

## 5.4. Режимы игр

### 5.4.1. Одиночный режим

Одиночный режим должен быть доступен без подключения к серверу.

Предполагаемые одиночные игры:

- Слоты
- Очко
- при необходимости локальная рулетка

### 5.4.2. Многопользовательский режим

Многопользовательские игры должны работать только при подключении к серверу.

Многопользовательские режимы:

- Рулетка
- Дурак
- Покер
- Слоты в режиме комнаты

---

## 5.5. Комнаты

Система должна поддерживать комнаты для сетевых игр.

### 5.5.1. Общие требования

- комната создаётся под конкретную игру;
- комнаты отображаются в общем списке;
- комната может быть открытой или закрытой;
- закрытая комната требует пароль;
- владелец комнаты может управлять настройками;
- владелец комнаты может запускать игру;
- владелец комнаты может исключать игроков;
- в списке комнат должен отображаться замок для закрытых комнат.

### 5.5.2. Поля комнаты

- название комнаты
- игра
- владелец
- текущее количество игроков
- максимальное количество игроков
- статус комнаты
- открытая/закрытая
- пароль (если закрытая)

### 5.5.3. Статусы комнаты

- ожидание игроков
- игра началась
- завершена

---

## 5.6. Подключение к серверу

### 5.6.1. Сервер

- сервер запускается на ноутбуке;
- используется только локальная сеть;
- подключение через IP-адрес сервера;
- отдельный внешний хостинг не требуется.

### 5.6.2. Требования к сетевому режиму

- при отсутствии подключения к серверу должен быть доступен локальный режим;
- при наличии подключения становятся доступны сетевые игры и список комнат.

---

## 5.7. Игровые требования по модулям

## 5.7.1. Слоты

### Функциональность

- одиночная игра;
- возможность играть через комнату;
- настройка ставки;
- анимация прокрутки;
- расчёт выигрыша;
- списание/начисление баланса.

### UI

- визуальные барабаны;
- анимация вращения;
- эффект выигрыша/проигрыша.

---

## 5.7.2. Очко

### Функциональность

- игра по классическим правилам очка;
- одиночный режим;
- ставки;
- расчёт победы/проигрыша.

---

## 5.7.3. Рулетка

### Основной режим

- многопользовательская игра;
- несколько игроков делают ставки в одной комнате;
- после подтверждения всех ставок запускается один общий спин;
- система рассчитывает результат для всех игроков.

### Правила

- европейская рулетка с одним зеро `0`;
- правила максимально близки к реальным.

### Ставки

- игрок может вводить количество фишек самостоятельно;
- все ставки должны требовать подтверждения перед спином;
- до подтверждения игрок может изменить или удалить ставку;
- интерфейс должен минимизировать миссклики.

---

## 5.7.4. Дурак

### Основной режим

- многопользовательская игра;
- до 4 игроков;
- колода 36 карт.

### Поддерживаемые режимы

- подкидной дурак;
- переводной дурак;
- выбор режима при создании комнаты.

### Правила

- правила должны быть максимально приближены к классическим;
- возможность подкидывания в соответствии с настройками и числом игроков;
- защита картами по обычным правилам;
- добор карт по правилам;
- завершение партии по классической логике.

### Дополнительно

- поддержка стикеров вместо текстового чата;
- стикер отображается над иконкой игрока примерно 2 секунды.

---

## 5.7.5. Покер

### Основной режим

- многопользовательский режим;
- Texas Hold’em;
- от 2 до 6 игроков.

### Поддерживаемые механики

- dealer button;
- small blind;
- big blind;
- передача блайндов по кругу;
- стадии: preflop, flop, turn, river;
- showdown;
- таймер хода;
- переподключение игрока в течение 20 секунд;
- при отсутствии переподключения — удаление из активной партии по логике стола.

### Действия игрока

- fold
- check
- call
- raise
- all-in

### Дополнительные требования

- поддержка side pots;
- логика ставок должна быть максимально похожа на реальную игру, но без лишнего усложнения интерфейса;
- дилер — система;
- поддержка стикеров вместо чата.

---

## 5.8. Стикеры

Система должна поддерживать быстрые стикеры в сетевых карточных играх.

### Требования

- набор предустановленных стикеров;
- при выборе стикер отображается у игрока над иконкой;
- время отображения — около 2 секунд;
- текстовый чат не требуется.

---

## 5.9. Анимации

Анимации должны присутствовать во всех частях приложения, где это уместно.

### Обязательные анимации

- переходы между экранами;
- вращение барабанов в слотах;
- вращение рулетки;
- раздача карт;
- перемещение фишек в покере;
- подсветка активного игрока;
- появление и исчезновение стикеров;
- визуальные эффекты победы/проигрыша.

---

## 5.10. Звуковые эффекты

### Общие требования

- использовать только звуковые эффекты, без фоновой музыки;
- пользователь должен иметь возможность включить/выключить звук.

### Примеры эффектов

- нажатие кнопок;
- раздача карт;
- вращение рулетки;
- вращение барабанов;
- фишки;
- победа/проигрыш;
- предупреждение о завершении таймера.

- сложность синхронизации состояния игр в сети;
- трудности с кроссплатформенной сборкой APK;
- возможные конфликты в совместной разработке.

### Способы снижения рисков

- сначала реализовать общий каркас;
- сначала реализовать простую рабочую версию, потом расширять;
- использовать GitHub-ветки;
- фиксировать формат данных и общие DTO заранее;
- писать модули независимо через общий интерфейс.

## 16. Перспективы развития

В перспективе проект может быть расширен за счёт:

- новых игр;

- таблицы лидеров;
- более гибких настроек комнат;
- улучшенной анимации и визуальных эффектов.

## 17. Итоговая формулировка проекта

Разрабатываемое приложение представляет собой кроссплатформенную клиент-серверную игровую систему на Python/Flet с поддержкой локального и сетевого режима, общей системой аккаунтов, сохранением баланса и статистики, списком комнат и набором карточных и казино-игр, реализованную с акцентом на качественную объектно-ориентированную архитектуру.
